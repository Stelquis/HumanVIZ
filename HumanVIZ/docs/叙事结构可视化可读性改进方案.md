# 叙事结构可视化可读性改进方案

## 一、现状问题诊断

通过对 [p4_opera_ribbon_viewer.tsx](../src/components/TaskViews/p4_opera_ribbon_viewer.tsx)、[Task4Layout.tsx](../src/components/TaskViews/Task4Layout.tsx)、[StoryCurve](storycurve-master/src/storycurve.js) 及 [p4_story_ribbon_core.ts](../src/utils/p4_story_ribbon_core.ts) 的深度分析，当前叙事结构可视化存在以下七个核心可读性问题：

### 1.1 单维度平面布局导致视觉"意大利面条"效应

当前所有信息（角色丝带、阶段色带、冲突曲线、场景背景、分隔线、标签）堆叠在同一 2D 平面中。角色丝带占据主要视觉空间，而叙事结构信息（阶段划分、冲突曲线）被挤压在边缘。冲突曲线仅占据 50px 的窄带区域（[p4_opera_ribbon_viewer.tsx:754](p4_opera_ribbon_viewer.tsx#L754)），在视觉上几乎不可察觉。

### 1.2 叙事阶段划分采用硬编码比例，缺乏数据驱动

[NARRATIVE_PHASES](p4_opera_ribbon_viewer.tsx:699-704) 对所有剧本使用固定百分比 [0-20%-55%-80%-100%] 划分开端/发展/高潮/结局。但《赵氏孤儿》存在跨代"转折"阶段、《贵妃醉酒》以情感三阶递进——固定划分完全忽略了剧本的实际叙事节奏。

### 1.3 StoryCurve 双轴映射核心理念完全未被利用

[StoryCurve](storycurve-master/src/storycurve.js) 的核心创新在于同时映射"叙事顺序"（X 轴）和"故事顺序"（Y 轴），揭示非线性叙事结构。当前实现虽在注释中标注了 "storycurve-inspired"，但仅借鉴了背景色带和分隔线的表面视觉元素，**完全没有使用双轴映射**——这是最大的理念错失。

### 1.4 缺乏"概览→细节"的层次化信息架构

所有信息一次性全部渲染在单个 SVG 中，没有 Zoom/Pan、没有 Overview+Detail、没有按需展开的信息层。用户面对《赵氏孤儿》23 角色 × 大量场景的密集丝带时，完全没有办法聚焦局部。

### 1.5 角色文本标签可读性差

[角色名标签](p4_opera_ribbon_viewer.tsx:549-568) 使用 `rotate(90, ...)` 旋转，竖排文字可读性较差。同时场景标签也存在同样的问题。

### 1.6 情感/冲突信息编码单薄

角色在场景中的情感状态（`rating` 字段，-1 到 1）仅通过 marker 圆点的大小编码，无法传递情感极性（正面/负面）、情感变化趋势等关键信息。`positions.ts` 中实际已支持 `emotionPos` 情感轴布局（[positions.ts:307-330](positions.ts#L307)），但 p4 渲染层完全未使用。

### 1.7 缺乏跨剧本的叙事结构对比视图

当前仅支持单剧本查看，无法在同一个视图中对比两部剧本的叙事结构差异。Task4Layout 中虽然有五部代表性剧本，但用户需要在它们之间手动切换，缺乏并排比较的能力。

## 二、改进方案

### 方案一（核心推荐）：StoryCurve 双轴映射 + 自适应阶段检测的三层叙事结构图

**创新点**：将 StoryCurve 的故事顺序/叙事顺序双轴映射理念，与 StoryRibbon 的角色出场丝带结合，形成一种全新的"叙事空间图"。同时用数据驱动的自适应算法替代硬编码阶段划分。

**具体设计**：

```
┌─────────────────────────────────────────────────┐
│  Layer 1 (背景): 自适应叙事阶段色带                │
│  ┌──开端──┬────发展────┬──转折──┬─高潮─┬─结局─┐   │
│  │ 朱砂红 │  琉璃金    │ 云水青 │ 琥珀 │ 石板灰│   │
│  └────────┴───────────┴───────┴─────┴──────┘   │
│                                                  │
│  Layer 2 (中景): 场景事件节点 + 角色共现热力       │
│  ●────●──●─────●────●───●────●───●              │
│  │节点大小=重要性│节点颜色=情感极性 │             │
│                                                  │
│  Layer 3 (前景): 角色叙事丝带 (保留现有设计)        │
│  ═══诸葛亮══════════════════════════              │
```

**技术实现路径**：

1. **数据驱动阶段检测**：在 `p4_story_ribbon_core.ts` 中新增 `detectNarrativePhases()` 函数，利用冲突弧 `conflictArc` 和情感弧 `sentimentArc` 的局部极大值/极小值自动检测阶段边界，替代 [NARRATIVE_PHASES](p4_opera_ribbon_viewer.tsx:699-704) 的固定百分比。

2. **Y 轴重新映射**：将当前仅用于角色堆叠的 Y 轴改为双编码——下半部分继续用于角色丝带，上半部分用于"叙事空间图"的场景节点（Y 轴对应冲突/情感强度），真正实现 StoryCurve 的双轴映射理念。

3. **三层 SVG group** 渲染隔离：`<g id="layer-phase-bands">` → `<g id="layer-scene-nodes">` → `<g id="layer-character-ribbons">`，层间使用不同的透明度/混合模式。

**可行性**：
- 自适应阶段检测可直接复用现有的 `conflictArc`、`sentimentArc` 数据 —— 数据已就绪
- 三层渲染不改变现有丝带计算逻辑，仅调整 SVG 布局和坐标空间
- 参考 `positions.ts` 中已支持 `emotionPos`、`promPos` 等多种 Y 轴布局模式，基础设施完备

**预期改进**：叙事阶段划分准确率提升（从固定比例变为数据驱动）、冲突曲线从 50px 窄带扩展为独立视觉层、用户可同时观察宏观叙事结构与微观角色出场。

**具体设计**：

**技术实现路径**：

- 情感数据已存在于 `scene_data[].characters[].rating` 中

- 右侧面板有充足空间（[Task4Layout.tsx:542](Task4Layout.tsx#L542) 的 `t4-side-panel`）

**预期改进**：用户可一眼看出每个角色在整个故事中的情感轨迹，而不需要逐场景追踪丝带粗细变化。

## 四、核心技术细节

### 4.1 自适应阶段检测算法

```typescript
// 在 p4_story_ribbon_core.ts 中新增
export interface NarrativePhase {
  label: string;
  startScene: number;
  endScene: number;
  dominantFeature: 'conflict' | 'sentiment' | 'density';
}

export function detectNarrativePhases(
  scenes: Scene[],
  conflictArc: number[],
  sentimentArc: number[],
  characterDensity: number[]
): NarrativePhase[] {
  // 1. 计算 conflictArc 的转折点（局部极大值 → 高潮候选）
  const conflictPeaks = findLocalExtrema(conflictArc, 'max');
  
  // 2. 计算 sentimentArc 的变化率（大斜率跳跃 → 转折候选）
  const sentimentSlopes = sentimentArc.map((v, i) => 
    i === 0 ? 0 : Math.abs(v - sentimentArc[i - 1])
  );
  const sentimentBreakpoints = findLocalExtrema(sentimentSlopes, 'max');
  
  // 3. 计算 characterDensity 的骤变点（群戏切换 → 阶段候选）
  const densityChanges = characterDensity.map((v, i) =>
    i === 0 ? 0 : Math.abs(v - characterDensity[i - 1])
  );
  const densityBreakpoints = findLocalExtrema(densityChanges, 'max');
  
  // 4. 融合三类断点，取 top-N 个作为阶段边界
  const allBreakpoints = mergeBreakpoints(
    conflictPeaks, sentimentBreakpoints, densityBreakpoints
  );
  
  // 5. 根据断点特征自动标注阶段类型
  return classifyPhases(allBreakpoints, scenes, conflictArc, sentimentArc);
}
```

### 4.2 波形图与丝带图的对齐机制

波形图和丝带图共用 `scenePos` 数组（[positions.ts:145](positions.ts#L145)），确保两者在每个场景位置 X 坐标上精确对齐。波形图区域使用独立的 Y 比例尺（0-1 冲突/情感值），丝带图区域继续使用现有的角色 Y 轴布局。

## 五、风险与注意事项

1. **角色数量极端差异**：《连环套》30 角色 vs 《贵妃醉酒》3 角色——波形图高度应设为百分比而非固定像素，避免小剧本中波形图占比过大。
2. **自适应检测的边界情况**：单幕剧或过短剧本（≤3 场）可能无法提取有意义的阶段，需保留固定比例作为 fallback。
3. **性能**：三层 SVG 渲染 + 小倍数图可能增加 DOM 节点数，建议对角色数 >15 的剧本限制小倍数图的角色数量（只展示 top-8 角色）。
4. **不与现有代码冲突**：所有改动应在 `p4_opera_ribbon_viewer.tsx` 中通过 props 可选控制（如 `enableWaveform?: boolean`），确保向后兼容。

## 六、StoryCurve 理念的深层借鉴

当前实现仅借鉴了 StoryCurve 的**表面视觉元素**（背景色带、分隔线），但 StoryCurve 真正的核心贡献在于：

1. **双轴语义映射**：X = Narrative Order（叙事顺序），Y = Story Order（故事时间顺序）。对于京剧剧本，这可以映射为 X = 场次顺序，Y = 剧情时间线（如《赵氏孤儿》跨 15 年的时间跳跃在 Y 轴上产生大的位移）。

2. **场景元数据叠加**：StoryCurve 的 `band`（地点）和 `backdrop`（时间）背景层，在京剧场景中可映射为"舞台场景类型"（宫殿/战场/街市/山林）和"时间"（日/夜/晨/暮）。

3. **Zoom/Pan 交互**：StoryCurve 内置的缩放和平移能力（[storycurve.js:443-458](storycurve-master/src/storycurve.js#L443)），对处理《赵氏孤儿》《连环套》等长剧本极为必要。

4. **Children 堆叠布局**：StoryCurve 将每个场景的子元素（角色）垂直堆叠（[storycurve.js:200-208](storycurve-master/src/storycurve.js#L200)），这种紧凑布局对角色密度高的场景有更好的空间利用率。

**建议**：在方案一的"叙事空间图"实现中，将这些 StoryCurve 核心理念系统性地融入，而非仅停留在视觉元素层面的模仿。
