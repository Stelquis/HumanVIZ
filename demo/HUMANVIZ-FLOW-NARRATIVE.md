# HumanVIZ 叙事文本智能解析系统

> 基于大语言模型的文学文本结构化分析与可视化数据生成流程

## 一、系统架构与运行环境初始化

HumanVIZ 是一套面向叙事文本的智能化解析与数据生成系统，旨在通过大语言模型的深层语义理解能力，将非结构化的文学文本转化为高维结构化的可视化数据。系统以 Jupyter Notebook 形态交付核心处理模块，采用多阶段流水线架构，融合异步并行计算与结构化输出控制，在保证数据质量的前提下实现高效的批处理能力。

系统启动阶段首先完成运行环境的配置与依赖注入。技术栈以 LangChain 作为大语言模型调用框架，借助其统一的接口抽象实现多模型Provider的无缝切换；Pydantic 承担数据建模与校验职责，通过类型安全的模式定义确保模型输出的结构一致性；asyncio 与 aiofiles 构建异步IO能力层，为高并发场景提供性能基础。API 层采用 MIMO 平台的 OpenAI 兼容接口，通过直接明文配置 API 密钥、模型标识与基础端点三个核心参数，消除了传统密钥文件管理的复杂性，显著提升了部署效率与调试便捷性。工作目录体系遵循功能分离原则，scripts 目录承载原始输入文本，json 目录持久化中间处理状态，chapters 目录维护章节级切片数据，形成清晰的输入-处理-输出数据流。

数据模型层是系统正确性的核心保障。StoryInfo 模式定义故事级元数据的提取规范，涵盖章节总量、文本边界标记等全局信息；SceneInfo 与 SceneListInfo 模式约束场景切分阶段的输出结构，精确控制场景标题、叙事摘要与行列坐标的生成；CharacterInfo 模式深入刻画角色实体的多维属性，包括身份标识、叙事重要性、情感极性、情绪状态与代表性引文；SceneDetails 模式整合场景上下文信息，关联空间地点、叙事张力与角色阵容；ChapterSummary 与 CharacterLinks 模式支撑更高阶的叙事分析，生成章节概述与角色关系图谱。这些模式定义通过描述性元数据与类型约束，构建起从自由文本到结构化数据的转换桥梁。

## 二、文本预处理与章节级粗粒度分割

预处理阶段的首要任务是从配置层加载故事元数据并建立输入输出通道。系统要求用户显式声明故事标识符、分析范式（角色中心或主题中心）以及作品元信息（题名、创作者、体裁、封面资源）。输入模块从 scripts 目录加载对应标识符的原始文本，同时支持从既有 JSON 结果中恢复处理状态，实现断点续处理与增量更新。针对多卷本长篇作品，系统内置卷册识别逻辑，自动定位 END OF VOL 或 VOLUME II 等分隔标记并截取首卷内容，确保处理范围的叙事完整性。

章节分割作为粗粒度内容切分环节，面临大语言模型上下文窗口的物理限制。系统采用头部-尾部联合采样策略，当文本长度超过十二万字符阈值时，提取开篇与结局各六万字符构建代表性样本，而非机械式顺序切块，从而在控制 Token 消耗的同时保留叙事的起承转合特征。随后调用 StoryInfo 结构化模型，提示大语言模型识别章节边界标记。该过程要求模型输出严格符合 JSON Schema 的响应，包含章节计数、文本首行、末行以及完整的章节标记列表。系统通过异常捕获-重试循环机制处理模型输出的不确定性，对解析失败或格式偏差的情况自动触发重新推理，直至获得有效响应。

章节标记获取后进入精确分割阶段。文本首先经过规范化清洗，包括空白符合并、换行符统一、智能引号转换等预处理操作。系统利用首行内容与首章标记定位叙事起点，依据末行内容定位叙事终点，裁剪出有效文本区间。继而遍历章节标记序列，通过子串匹配确定各章节的起止字节位置，提取章节正文。为满足下游行号寻址需求，系统在每行文本前注入 LINE 序号标记，建立可追踪的行列坐标系。分割结果同步写入 chapters 目录与前端 public 目录，前者服务于后续处理阶段，后者直接为可视化界面的原文引用提供数据源。

## 三、细粒度场景解析与异步并发处理

场景分析构成系统计算负载最密集的环节，涉及场景边界识别、场景属性提取与角色信息抽取等多维任务。考虑到典型叙事作品包含数十章节、每章内含若干场景的规模特征，串行处理策略将导致不可接受的响应延迟。系统因此引入 asyncio 异步编程模型，构建高并发处理流水线。

并发控制层通过信号量机制限制并行任务数量，默认配置为二十并发，可根据 API 速率限制与服务容量动态调整。场景分析拆解为两个顺序依赖的子阶段：场景分割与场景详情提取。场景分割阶段对每个章节独立调用大语言模型，基于用户配置的分割策略（地点变更触发或角色出入触发）识别场景边界。模型输出场景元数据列表，各元素包含场景标题、一句话摘要、起始行号、结束行号以及分割依据说明。系统对模型输出执行有效性验证，剔除行号倒挂或越界的异常项，按起始行号排序后形成规范化的场景序列。

场景详情提取阶段面临更高的计算复杂度。对每个有效场景，系统调用 SceneDetails 模型执行深度分析，提示工程要求模型提取空间地点、叙事重要性评分、冲突强度评分以及角色阵容列表。针对每个识别角色，模型需输出身份标识、叙事重要性、重要性依据、情绪标签、情感极性量化值、原文代表性引文以及引文缺失时的替代说明。该阶段对提示词设计有严苛要求，系统通过精细化指令约束模型输出连续原文片段而非概括性描述，并过滤无名角色与功能性龙套，聚焦于具有叙事意义的角色实体。

异步架构在此阶段的价值得到充分体现。系统为每个章节创建独立协程任务，章节内部的多个场景并行处理，通过 asyncio.gather 聚合全部结果。并发处理理论上可将总耗时从串行模式的数十分钟压缩至数分钟级别。后处理阶段为每个场景计算全局重要性排名与冲突强度排名，建立跨章节的叙事张力比较基准。章节级结果持久化为独立 JSON 文件，全量场景合并至 all.json，为后续数据精修环节提供统一的数据视图。

## 四、实体去重与语义标准化

场景分析输出的原始数据存在实体冗余与命名异构问题。同一角色实体可能以全名、昵称、简称、代称等多种形态散布于不同场景，如 Robert、Bob、Bobby 指向同一人物；地点实体同理，Los Angeles、LA、City of Angels 等指称共享同一地理实体。未进行标准化处理的数据将扭曲角色统计结果，并在关系网络中生成重复节点。系统专门设计了实体对齐与消歧流程。

数据清洗首先构建全局实体词典。系统遍历 all.json 中的全部场景记录，抽取角色名称与地点名称，执行去重操作形成候选实体集合。随后构造分组任务提示词，要求大语言模型识别指向同一实体的名称变体集合。角色分组任务需建立标准名称到别名列表的映射，并严格区分相关但独立的角色（如 Bob 与 Bob's mother 应划分为不同条目）；地点分组任务遵循地理包含原则，将微观地点归并至其所属的宏观地点（如将 Bob's bedroom、Bob's garden 统一至 Bob's house）。

提示词模板采用少样本学习范式，通过示例输出阐明格式规范。分组任务选用 Claude 模型执行，利用其在复杂推理与指令遵循方面的优势。模型返回 JSON 格式的分组字典后，系统在全量场景中执行实体名称替换。替换策略结合精确匹配与整词匹配，运用正则表达式避免子串误匹配（如确保 Mitsuha 不会错误匹配至 Mitsuha's），同时妥善处理所有格形态。实体名称执行首字母大写的形态规范化，确保命名一致性。清洗后的纯净数据回写至 all.json，角色与地点实体实现语义对齐。

## 五、章节摘要生成与关系网络构建

章节摘要阶段从场景级数据向上抽取章节级叙事特征，包括整体摘要、角色互动拓扑与统计指标。该阶段延续异步并发架构，各章节的摘要生成任务独立调度。

单章处理流程首先聚合所属全部场景，拼接场景摘要形成该章的叙事轨迹。调用 ChapterSummary 模型生成一句话章节摘要，凝练核心叙事内容。同步构建角色共现网络：遍历章节内全部场景与角色，当两个角色在同一场景出现时建立边连接，边权重累加共现场景数。对于存在连接的每对角色，调用 CharacterLinks 模型基于场景摘要生成互动描述，刻画该章节中的具体关系性质。

统计维度涵盖场景计数、文本长度、行数、平均重要性、平均冲突强度，以及地点与角色的频次分布。章节间执行重要性排序与冲突强度排序，标识叙事的关键转折点与高潮段落。章节摘要数据结构包含章节标识、场景统计、文本摘要、重要性评分、冲突评分、全局排名、地点分布、角色分布与关系链接等多元字段，构成故事叙事结构的数字化描述。

## 六、角色画像构建与可视化属性分配

角色摘要阶段为每个唯一角色构建丰富的画像信息。受限于大语言模型的上下文容量，当角色数量超出阈值时系统采用分批处理策略，每批上限二十个角色，筛选包含目标角色的场景片段作为上下文输入。

CharacterInfo 模式要求模型提取代表性引文、功能分组与可视化颜色。代表性引文应体现角色性格特征或叙事地位；功能分组将角色按叙事功能归类（如主角阵营、对立阵营、辅助阵营），要求组间互斥且每组至少包含两个成员；可视化颜色分配要求选择具有区分度与审美价值的 RGB 值，并提供选色依据说明，颜色需在白色背景下保持可见性。

颜色分配任务对一致性要求较高，系统仅对出场频次最高的前二十位角色执行该步骤，其余角色留空。颜色分配完成后执行分组名称标准化，将语义相近的分组标签归并统一（如 Main Characters、Protagonists、Heroes 统一映射至主角组）。最终角色数据结构包含名称、引文、分组、颜色与解释五个属性，支撑前端角色卡片与图例渲染。

## 七、地点画像生成与最终数据整合

地点摘要遵循简化的处理逻辑。系统为每个唯一地点提取代表性描述引文与符号化表情。表情符号的选择需直观传达地点特征，如城堡对应城堡图标、森林对应树木图标。地点信息组织为包含名称、引文与表情的记录结构。

最终数据整合阶段将全部分析结果汇拢为统一的 JSON 数据结构。根层级包含作品元数据（题名、体裁、创作者、年份、封面资源）与宏观统计（章节数、场景数、角色数、地点数）。chapters 数组存储章节摘要集合，scenes 数组存储场景详情集合，characters 数组存储角色画像集合，locations 数组存储地点信息集合。该结构即为 HumanVIZ 前端可视化引擎的标准输入格式。

系统执行双路径持久化：json 目录下的 final_data.json 作为分析结果的归档备份，src/data 目录下的命名文件供前端应用直接加载。至此，从原始叙事文本到可视化就绪数据的完整转换流程执行完毕，用户可通过网络图、时间轴、热力图等多维视图探索故事的叙事结构与角色关系。

## 八、分析范式与增量更新机制

系统支持角色中心与主题中心两种分析范式。角色模式下分析对象聚焦于人物实体及其属性；主题模式则将分析目标转向叙事主题概念，提取核心主题、主题关联与主题情感色彩。两种范式共享相同的处理流程骨架，差异仅体现在提示词设计与输出模式定义，使系统能够适配多元化的文学分析需求。

增量更新机制支撑迭代式分析工作流。当用户已完成完整场景分析但需调整角色或主题提取策略时，可将 scene_details_exist 标志置为 True 进入增量模式。该模式下系统直接加载既有 all.json，仅执行角色或主题信息的重新提取与替换，规避耗时的场景分割与详情提取阶段，显著缩短迭代周期。此机制对于分析策略调优与范式切换具有重要实用价值。

全流程执行耗时取决于文本规模与场景数量，典型中篇小说（约五至十万字）的处理时间介于三至十分钟。系统记录处理起止时间戳并输出总耗时指标，供性能评估参考。用户可通过调节并发参数、选择分析范式在效率与质量之间取得平衡，适配不同场景的需求约束。
