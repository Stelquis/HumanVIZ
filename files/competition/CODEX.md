# ChinaVis 2026 赛道 1-I 答卷编写约定

本文件用于约束后续所有答卷修改。每次改写 [papre.tex](/workspace/files/competition/papre.tex) 前，先阅读本文件，再开始回答或编辑。

## 1. 总目标

- 答卷要服务于评分，不是产品介绍，也不是系统宣传。
- 写作目标是让评委快速看清：`你回答了什么问题、给出了什么结论、证据在哪里`。
- 优先保证“答对了多少”被清楚看见，而不是“系统做得多完整”。

## 2. 写作视角

- 站在“我们团队完成分析后的判断”来写。
- 口吻应当像大三的数据科学与大数据技术专业学生提交的分析答卷。
- 不要使用过强的模板化标签，如“总体回答：”“结论：”这类机械口吻。
- 优先使用自然表达，如：
  - `根据统计结果，我们认为……`
  - `从比较结果来看……`
  - `综合来看……`

## 3. 结构原则

- 每一题都遵循：`总起 -> 分解 -> 总结`
- 每一题都要先直接回答问题，再展开分类、阶段、模式或规律。
- 优先使用 `x.x` 编号结构，让评分点清楚可见。
- `x.x` 标题要描述内容本身，不要把“直接回答”这种答题动作写进标题里。

推荐结构：

- `x.1 分析思路 / 分析路径`
- `x.2 分类/模式拆解`
- `x.3 阶段/规律拆解`
- `x.4 或 x.5 综合判断`

任务一已验证有效的开头/结尾写法：

- 开头第一段更适合写“我们如何处理数据、如何得到判断”，不必一上来就写“我们认为……”
- 小结更适合写成“综合来看，我们认为……”，不要写“本题的主要结论……”
- 当一段同时包含“处理过程”和“统计发现”时，应拆成两段：先过程，后结果

## 4. 表格、列表与公式原则

- 因为每题有 800 字建议限制，所以优先用表格、列表或公式承载信息，但三者都不是越多越好。
- 形式的任务是“正面回答问题”，不是装饰。
- 每个表格、列表或公式都应直接对应一个分类、阶段、模式或指标比较点。
- 只有能提高评分效率的内容才格式化；纯过渡句、解释性长句不要硬塞进表格。
- 不把整题开头做成“问题 / 我们的回答”大表；题目开头优先用自然语言总起，展开部分再按内容选择表格、列表或公式。
- 正文中先判断内容是否并列或递进，再决定用无序列表、有序列表还是表格。
- 统计结论可直接写检验量、相关系数、显著性，不必为了“表格化”再重复一遍。
- 统计指标第一次出现时，要写中文含义，不只堆英文或缩写。
- 表格外观要简洁，优先避免满框线；表头短、列少、信息密度高。

推荐表头风格：

- `类别 / 特点 / 解释`
- `阶段 / 主要现象 / 说明`
- `模式 / 结构特征 / 常见剧种`
- `主题组合 / 主导剧种 / 说明`

推荐列表风格：

- 先给总判断，再用 2-4 个条目拆解
- 每个条目只说一个层次的结论
- 需要时可用公式给出统计关系或检验结果
- 有先后、阶段、递进关系时用有序列表
- 并列类别、并列特征、并列发现时用无序列表
- 列表优先用于同级并列项，不把列表写成重复句式堆叠

常用判断：

- 需要分类归纳时，优先列表
- 需要显示对比关系时，优先表格
- 需要强调统计支撑时，优先公式和一句解释
- 需要说明阶段演变时，优先有序列表
- 需要举典型角色说明时，优先单独成段，不和总体统计硬挤在同一段里

避免使用：

- `最强对应证据`
- `第一层答案`
- `最终问题`
- 其他明显像 AI 模板或技术报告的措辞

## 5. 证据组织原则

- 证据必须以可视化图表为主。
- 允许使用单张图，也允许使用多图拼图。
- 因为图片数量有限，优先考虑拼图。
- 不使用文字截图作为主要证据。
- 文字说明直接写在答卷中，不靠截图展示文字。

证据链表达方式：

- 先给结论
- 再解释原因
- 再指出由哪些图支撑

图片注释规范：

- 每段证据建议都使用统一注释字段，便于后续截图、命名和排版。
- 推荐注释格式：
  - `图片编号`
  - `建议文件名`
  - `图型`
  - `放置位置`
  - `对应文字`
  - `截图要求`
- 建议文件名统一使用：
  - `q1-1-overview-collage`
  - `q1-2-role-case-guanyu`
  - `q1-3-period-trend-collage`
  - 依此类推，规则为：`q题号-段号-语义名`
- 拼图文件名统一以 `-collage` 结尾，单图文件名直接写主题名。
- 注释内容要做到一眼能看懂，不写模糊表述，如“这里放个图”“之后再截”。
- 图片注释最好直接写成可取消注释即用的 LaTeX `figure` 模板。
- 文字在前，图片在后；不要先放图再补结论。
- 除拼图外，不要把多张证据图连续堆在一起；总体图与个案图应分开放置。
- 图的证据强度要与正文措辞匹配：示意性对比图不要写成“严格统计结论”的唯一依据。

今天新增的证据经验：

- 图片不是越多越好；如果正文中的表格、列表和统计结果已经足够回答某一部分，就不要为了“凑满图片数”硬加图。
- 截图方案必须先服从前端真实可操作性，再谈排版理想效果。
- 优先使用页面中央主视图中的核心图表；谨慎使用窄侧栏、底部小卡片、弹窗边角等不利于阅读和截图的区域。
- 如果某张图需要依赖很小的侧栏区域、解释成本过高，或实际截出来不清楚，就应当删掉这张图，而不是勉强保留。
- 当一类证据已经由表格和统计量承担时，可以减少对应图片数量，把图片留给更直观、更难用文字替代的部分。
- 图片注释中的操作步骤必须与真实页面完全一致，不能写“理论上可以这样截”的方案。
- 若证据方案收缩，例如从 3 张改为 2 张，正文、图号、文件名、图注说明都要同步更新，不能只删图片不改文字。

任务四已验证有效的取图经验：

- 单剧分析优先截中央主图，不再依赖右侧窄栏中的辅助组件。
- 多剧比较优先截中央叠加曲线主图，不强行使用解释性较弱的附属比较图。
- “典型模式”这类信息若已由表格完整承担，可以不再单独配置第三张图。

## 6. 系统概览写法

- 第 0 部分是真正的“系统概览”，不是“我们如何写答卷”。
- 不写元话语，例如：
  - `本文按照什么组织`
  - `我们如何回答问题`
  - `下面将从……展开`
- 第 0 部分只写：
  - 系统名称
  - 面向的数据范围
  - 系统整体架构
  - 各层做什么
  - 总览页和任务页如何组织

当前系统名称统一为：

- `梨园万象——京剧剧本多维可视分析系统`

## 7. 标题与层级原则

- `系统概览`
- `1、`
- `2`
- `3、`
- `4、`
- `5、`
- `总结`

以上都属于同一级标题，格式必须一致。

## 8. 官方题干原则

- 五道题的题干必须保留官方完整原文。
- 不得精简题干，不得改写题意。
- 后续所有压缩只能压正文，不能压题干。

## 9. 语言风格原则

- 语言应简洁、明确、可评分。
- 少写泛泛空话，如：
  - `构建了完整链路`
  - `形成了良好支撑`
  - `体现出较强能力`
- 每一句尽量对应一个可检查的事实、判断或证据。
- 能拆成两句就不要硬塞成一个长句。

## 11. 修改前检查清单

每次回答或改稿前，先检查：

- 这段文字是在回答问题，还是在介绍系统？
- 是否先给出了明确结论？
- 是否能改成表格或更短的 list？
- 是否使用了过于生硬、模板化的表达？
- 是否给出了可视化证据位置？
- 这个证据位置在真实前端里是否容易截图、容易读懂？
- 如果删掉一张图，现有表格、列表和统计结果是否已经足够支撑这一段？
- 是否保持了“总起 -> 分解 -> 总结”的结构？
- 是否仍然站在“我们团队的分析判断”视角？

## 12. 编译命令

单次编译：

```bash
cd /workspace/files/competition && xelatex -interaction=nonstopmode -halt-on-error papre.tex
```

建议最终版编译两次：

```bash
cd /workspace/files/competition && xelatex -interaction=nonstopmode -halt-on-error papre.tex && xelatex -interaction=nonstopmode -halt-on-error papre.tex
```
